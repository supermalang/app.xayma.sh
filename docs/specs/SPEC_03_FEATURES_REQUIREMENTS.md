## 1. Core MVP Features (Blockers)

These 5 features must work for the product to be viable at launch:

### F1 — Authentication & Role-Based Access
- Supabase Auth (email/password)
- JWT session management
- Role-based UI and API access (ADMIN, CUSTOMER, RESELLER, SALES, SUPPORT)
- Protected routes per role
- **Done when:** Each role sees only their permitted views; unauthorized access returns 403; JWT refresh works seamlessly

### F2 — Partner & User Management
- CRUD for partners (customers, resellers, affiliates)
- CRUD for users linked to partners
- Partner status management (active, suspended, low_credit, etc.)
- Partner type classification (customer, reseller, pro_reseller, affiliate)
- **Done when:** Admin can create/edit/suspend any partner; users linked correctly; all partner types behave correctly in the UI

### F3 — Service & Deployment Management
- Service catalogue (Odoo Community, custom Docker)
- Service plan configuration (credit consumption per plan)
- Deployment CRUD with AWX integration
- Real-time deployment status via Supabase Realtime
- Control node management
- **Done when:** Admin can configure a service; customer can deploy it; AWX job runs; status updates in real time; domain is routable via Traefik

### F4 — Credit System with Kafka
- Credit purchase flow (Payment Gateway API integration)
- Credit deduction every 15 minutes via Kafka → n8n → Supabase
- Credit balance display in real time (Supabase Realtime/WebSockets)
- Low credit alerts (20%, 10%, 0%)
- Grace period management per partner type
- Reseller volume discount tiers (10/20/40 instances)
- Credit expiry (30/45/60 days per bundle size)
- **Done when:** Credits deduct accurately every 15 min; alerts fire at correct thresholds; expired credits removed; Kafka events logged in audit

### F5 — Notification System
- WhatsApp (via n8n + WhatsApp Business API or Twilio)
- Email (via n8n + SMTP/SendGrid)
- SMS (via n8n + Africa's Talking or similar)
- In-app notifications (Supabase Realtime → Vue store)
- **Done when:** All 4 channels deliver for credit warnings, deployment events, and payment confirmations

### F6 — Voucher System

A voucher is a single-use code generated by an Admin that adds a fixed number of credits to a partner account on redemption. Primary use case: free trial offers, sales promotions, beta customer onboarding.

**Admin capabilities:**
- Generate one voucher or a batch of vouchers (up to 100 at once)
- Set credit amount per voucher (any integer, e.g. 10 = 1 month Starter free)
- Set optional expiry date
- Optionally restrict to a specific `partner_type` (e.g. customer only)
- Optionally assign to a specific partner (single-use personal voucher)
- View all vouchers with status (active / redeemed / expired)
- Deactivate a voucher before redemption

**Customer/Reseller capabilities:**
- Enter voucher code on the Buy Credits page ("Redeem a voucher" tab)
- Immediate credit top-up on valid redemption — same Realtime update as a payment
- Error messages for: invalid code, already redeemed, expired, wrong partner type

**Business rules:**
- One redemption per voucher per partner (even if `max_uses > 1`, a partner cannot redeem the same code twice)
- Redemption recorded as a `credit_transaction` with `transactionType = 'credit'` and `paymentMethod = 'voucher'`
- Voucher redemption publishes `credit.topup` to Kafka — same downstream flow as a payment (resumes suspended deployments, triggers notifications)
- Expired vouchers cannot be redeemed regardless of status
- Admin can set `max_uses = 1` (personal voucher) or `max_uses > 1` (broadcast code, e.g. launch campaign)

**Done when:** Admin generates a voucher; customer redeems it; credits appear in real time; second redemption by same partner is blocked; expired code is rejected with clear error message.

## 4. Acceptance Criteria per Core Feature

### F1 — Auth
- [ ] Login with email/password succeeds and returns JWT
- [ ] Invalid credentials return clear error message
- [ ] Each role redirected to correct dashboard on login
- [ ] Route guard blocks unauthorized access and redirects to login
- [ ] Session persists across browser refresh
- [ ] Logout clears session and redirects to /login

### F2 — Partner Management
- [ ] Admin can create partner with all required fields
- [ ] Partner slug auto-generated from name
- [ ] Phone number validated against West African format (70/75/76/77/78 XXXXXXX)
- [ ] Partner status changes cascade to deployment status
- [ ] Reseller partner shows volume discount options
- [ ] Audit log entry created on every partner mutation

### F3 — Deployments
- [ ] Deployment creation triggers AWX job within 10 seconds
- [ ] Status transitions: pending_deployment → deploying → active (or failed)
- [ ] Real-time status visible via Supabase WebSocket without page refresh
- [ ] Failed deployment shows error reason and retry button
- [ ] Stopped/suspended deployment shows resume option
- [ ] Domain name validated before submission

### F4 — Credits
- [ ] Payment gateway payment initiates correctly and returns callback
- [ ] Credits added to partner account within 60 seconds of payment confirmation
- [ ] Kafka credit.debit event published every 15 minutes per active deployment
- [ ] Credit balance in UI updates in real time without refresh
- [ ] Suspension triggered automatically when credits reach 0
- [ ] Grace period counter visible in dashboard
- [ ] Reseller discount applied correctly based on bundle size

### F5 — Notifications
- [ ] WhatsApp message delivered within 2 minutes of trigger event
- [ ] Email delivered within 5 minutes
- [ ] SMS delivered within 5 minutes
- [ ] In-app notification appears without page refresh
- [ ] Notifications respect language preference (FR/EN)

### F6 — Vouchers
- [ ] Admin generates a single voucher or batch (up to 100) with: credit amount, optional expiry date, optional partner_type restriction, optional max_uses
- [ ] Generated codes are unique, random, uppercase alphanumeric with dashes (e.g. `XAYMA-A3F9-K2LP`)
- [ ] Customer/Reseller enters code on Buy Credits page → credits added within 5 seconds
- [ ] Redemption creates a `credit_transaction` with `paymentMethod = 'voucher'`
- [ ] Same partner cannot redeem the same voucher code twice
- [ ] Expired voucher returns clear i18n error message
- [ ] Already-redeemed voucher (max_uses reached) returns clear i18n error message
- [ ] Wrong partner type restriction returns clear i18n error message
- [ ] Redemption publishes `credit.topup` to Kafka — resumes suspended deployments exactly like a payment
- [ ] Admin voucher list shows: code, credits, max_uses, uses_count, expiry, status, partner_type restriction
- [ ] Admin can deactivate any unredeemed voucher instantly

## 5. Compliance & Accessibility

- **WCAG:** AA target for all public-facing pages (marketing site)
- **Data privacy:** No PII stored outside Supabase (EU servers); GDPR-aware design
- **Payment security:** Payment Gateway handles card data; Xayma.sh never stores raw card numbers
- **Audit trail:** All mutations logged in `general_audit` table (who, what, when, old value, new value)
- **i18n:** All UI strings in FR + EN; language switchable at any time
