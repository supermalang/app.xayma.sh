# SPEC 08 — Deployment & Infrastructure
> Xayma.sh · v2 · Last updated: March 2026

---

## 1. Infrastructure Overview

| Node | Hetzner Plan | RAM | Role |
|------|-------------|-----|------|
| Management node | CX32 | 8GB | Traefik, deployment engine, workflow engine, Kafka, Strapi, Portainer, Datadog agent, Vue app container, Nuxt marketing container |
| Customer node(s) | CX52 | 32GB | Odoo customer instances (~20 per node) |
| Backup storage | Contabo VPS (large disk) | 4GB | Offsite backup target |

All nodes communicate via **Hetzner Private Network** (10.0.0.x range). Only Traefik exposes ports 80/443 to the public internet.

## 3. Docker Architecture

### Management Node — docker-compose.yml (production)
```yaml
services:
  traefik:         # Reverse proxy + SSL (Let's Encrypt)
  deployment-engine: # Infrastructure automation
  workflow-engine: # Workflow automation
  kafka:           # Event streaming (KRaft mode)
  kafka-ui:        # Kafka management UI (internal only)
  strapi:          # Headless CMS
  portainer:       # Docker management UI
  xayma-app:       # Vue 3 SPA (served by nginx)
  xayma-marketing: # Nuxt 3 SSR
  datadog-agent:   # Monitoring agent
```

### Customer Nodes — per deployment
```
Each customer instance = 2 containers:
  - odoo_<slug>     (Odoo Community app, 512MB–4GB depending on plan)
  - postgres_<slug> (PostgreSQL DB, 512MB–2GB depending on plan)

Resource limits set via Docker --memory and --cpus flags
Applied by deployment engine at provisioning time based on serviceplan
```

## 4. CI/CD Pipeline — GitHub Actions

### Workflow: `deploy.yml`
Trigger: Push to `main` branch

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - Checkout code
      - Set up Docker Buildx
      - Login to DockerHub (DOCKERHUB_USERNAME, DOCKERHUB_TOKEN secrets)
      - Build Vue app image → push to dockerhub/xayma-app:latest
      - Build Nuxt marketing image → push to dockerhub/xayma-marketing:latest

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - SSH to CX32 management node (SSH_PRIVATE_KEY secret)
      - docker pull xayma-app:latest && xayma-marketing:latest
      - docker compose up -d --no-deps xayma-app xayma-marketing
      - Health check: curl https://app.xayma.sh/health
      - Notify via Datadog event on success/failure
```

### GitHub Secrets Required
```
DOCKERHUB_USERNAME
DOCKERHUB_TOKEN
HETZNER_SSH_PRIVATE_KEY
HETZNER_CX32_HOST
SUPABASE_URL (build-time injection)
SUPABASE_ANON_KEY (build-time injection)
DATADOG_API_KEY
```

## 6. Monitoring — Datadog

| What | How |
|------|-----|
| Container metrics | Datadog agent Docker integration |
| Application logs | Datadog log collection from all containers |
| APM | Datadog APM on workflow engine and Nuxt (Node.js tracer) |
| Uptime checks | Datadog Synthetic monitors on app.xayma.sh and xayma.sh |
| Alerts | PagerDuty-style alerts → WhatsApp/email on: container down, CPU >80%, memory >80%, failed deployment |
| Kafka lag | Custom Datadog metric from kafka-ui API |
| Credit deduction accuracy | Custom metric: expected debits vs actual per 15-min window |
| Dashboard | Datadog custom dashboard: deployments, credits, revenue, errors |

### Key Alerts
```
- Container restart detected → immediate alert
- Kafka consumer lag > 100 messages → warning
- Workflow engine automation failed → immediate alert
- CX52 node memory > 85% → warning (time to add node)
- Payment IPN not received within 5 min of initiation → alert
- Failed deployment rate > 10% → alert
```

## 7. Backup Strategy

### Supabase (Database)
- Supabase Cloud automated daily backups (included in plan)
- Weekly manual export via `pg_dump` triggered by workflow engine cron → stored on Contabo
- Retention: 30 days

### Docker Volumes (Customer Data)
```bash
# Backup script — runs daily via cron on each CX52 node
#!/bin/bash
# Triggered by deployment engine scheduled job

for volume in $(docker volume ls -q | grep xayma_); do
  timestamp=$(date +%Y%m%d_%H%M%S)
  docker run --rm \
    -v $volume:/source:ro \
    -v /backup:/backup \
    alpine tar czf /backup/${volume}_${timestamp}.tar.gz -C /source .
done

# rsync to Contabo backup node
rsync -az /backup/ backup@contabo-node:/xayma-backups/$(hostname)/
```

### Backup Schedule
| Data | Frequency | Retention | Location |
|------|-----------|-----------|---------|
| Supabase DB | Daily (automatic) | 30 days | Supabase Cloud |
| Supabase DB | Weekly (manual pg_dump) | 90 days | Contabo |
| Customer Docker volumes | Daily | 14 days | Contabo |
| Workflow engine automations | Daily export | 30 days | Contabo |
| Kafka topics | Not backed up (transient) | N/A | N/A |
| Docker images | Permanent | N/A | DockerHub |

### Disaster Recovery
| Scenario | RTO | RPO | Procedure |
|----------|-----|-----|-----------|
| CX32 management node failure | 2h | 24h | Provision new CX32, restore from DockerHub + backup |
| CX52 customer node failure | 4h | 24h | Provision new CX52, restore volumes from Contabo |
| Supabase outage | 0 (Supabase handles) | 0 | Supabase SLA covers this |
| Accidental data deletion | 1h | 24h | Restore from pg_dump on Contabo |

## 8. Security Requirements

| Requirement | Implementation |
|-------------|---------------|
| HTTPS everywhere | Traefik + Let's Encrypt auto-renewal |
| Private network | All internal services on Hetzner private 10.0.0.x |
| Secrets management | Docker secrets / environment variables; never in code |
| Supabase RLS | All tables have row-level security enabled |
| Service role key | Only in workflow engine environment; never in browser |
| SSH access | Key-based only; password auth disabled on all nodes |
| Docker socket | Portainer access restricted to admin IP |
| Kafka | No public port exposed; internal network only |
| Deployment engine | Internal network only; accessed via Traefik with auth |
| Datadog API key | Stored as GitHub secret and Docker secret |
| Payment keys | Stored in Supabase settings table, accessed only by workflow engine |
| CORS | Vue app only; Supabase CORS restricted to app.xayma.sh |
